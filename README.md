# ТЕХНИЧЕСКОЕ ЗАДАНИЕ

# Проект: Развертывание VPN-инфраструктуры для пользователей в РФ

**Версия:** май 2026

---

# 1. ЦЕЛЬ ПРОЕКТА

Развернуть production VPN-инфраструктуру для пользователей на территории РФ с упором на:

- максимальную анонимность;
- устойчивость к DPI и блокировкам ТСПУ;
- стабильную работу через мобильные сети РФ;
- высокую скорость соединения;
- минимальный ping;
- централизованное удалённое управление;
- отказоустойчивость;
- масштабируемость;
- безопасность инфраструктуры.

---

# 2. ОСНОВНЫЕ ТРЕБОВАНИЯ

## 2.1 Анонимность и конфиденциальность

VPN-инфраструктура должна обеспечивать максимально возможный уровень анонимности пользователей и минимизацию цифровых следов.

### Обязательные требования

- полный отказ от логирования пользовательского трафика;
- не хранить:
  - IP-адреса пользователей;
  - историю подключений;
  - DNS-запросы;
  - историю посещений;
  - метаданные соединений;
- отключить телеметрию;
- отключить автоматическую отправку отчётов об ошибках;
- отключить отладочное логирование;
- использовать современные методы TLS-маскировки;
- минимизировать сетевой отпечаток VPN-трафика;
- использовать отдельные UUID/ключи для каждого устройства;
- реализовать регулярную ротацию ключей и точек подключения;
- минимизировать сигнатуры VPN-трафика.

---

## 2.2 Производительность

VPN должен обеспечивать максимально высокую скорость соединения и стабильную работу под нагрузкой.

### Требования

- поддержка минимум 10–15 устройств одновременно;
- высокая скорость соединения;
- минимальные потери пакетов;
- стабильная работа через мобильные сети РФ;
- быстрое восстановление соединения;
- автоматическое переключение между транспортными протоколами.

### Целевые показатели

- ping до российских ресурсов: не более 70 мс;
- потери пакетов: не более 1–2%;
- доступность сервиса (uptime): не менее 99%;
- отсутствие резких просадок скорости.

---

## 2.3 Совместимость с операторами РФ

VPN должен стабильно работать через:

- МТС;
- МегаФон;
- Билайн;
- Tele2;
- Yota.

### Требования

- стабильная работа через мобильный интернет;
- устойчивость к DPI;
- устойчивость к федеральным блокировкам;
- минимальное количество недоступностей;
- автоматическое переключение на резервный протокол при блокировке или деградации соединения.

---

## 2.4 Поддерживаемые устройства

VPN-клиент должен поддерживать:

- iPhone (iOS);
- Android;
- Windows;
- macOS.

### Дополнительные требования

- работа без root/jailbreak;
- корректная работа в фоне;
- автоматическое переподключение;
- безопасное хранение конфигураций и ключей.

---

# 3. РЕКОМЕНДУЕМЫЙ ТЕХНОЛОГИЧЕСКИЙ СТЕК

## 3.1 Основной production-протокол

# VLESS + XTLS-Reality + XHTTP

### Назначение

Основной VPN-протокол для всех пользователей.

### Причины выбора

- максимальная устойчивость к DPI;
- трафик выглядит как обычный HTTPS/TLS 1.3;
- отсутствуют стандартные VPN-сигнатуры;
- стабильно работает через мобильных операторов РФ;
- не зависит от UDP;
- устойчив к федеральным блокировкам.

### Требования

- использовать XTLS-Reality;
- транспорт:
  - основной: XHTTP;
  - резервный: gRPC;
- обязательная TLS-маскировка;
- использование TLS-отпечатков популярных браузеров;
- использование доменов с высокой репутацией;
- отдельный UUID на пользователя.

---

## 3.2 Высокоскоростной резервный канал

# Hysteria2 + Salamander

### Назначение

Высокоскоростной резервный канал для сетей без полной UDP-блокировки.

### Преимущества

- высокая скорость;
- низкий ping;
- эффективная работа на мобильных сетях;
- устойчивость к потерям пакетов;
- поддержка:
  - QUIC;
  - автоматической смены портов;
  - современного алгоритма управления перегрузкой канала.

### Ограничения

Может быть ограничен:

- на МТС;
- на МегаФон;
- при полной UDP-фильтрации.

### Требования

- обязательная Salamander-обфускация;
- автоматическое переключение на VLESS Reality при недоступности UDP;
- автоматическое переключение клиентов.

---

## 3.3 Дополнительный резервный протокол

# AmneziaWG

### Назначение

Дополнительный резервный протокол.

### Причины выбора

- менее детектируем по сравнению со стандартным WireGuard;
- высокая скорость;
- низкий ping;
- минимальные накладные расходы.

### Требования

- использовать только через AmneziaWG-обфускацию;
- стандартный WireGuard не использовать;
- автоматическая генерация конфигураций.

---

## 3.4 Исключённые технологии

| Технология | Причина |
|---|---|
| OpenVPN | легко определяется DPI |
| WireGuard | сигнатуры блокируются |
| L2TP/IPSec | устаревший протокол |
| PPTP | небезопасен |
| Shadowsocks/Outline | нестабильная работа |
| SOCKS5 без обфускации | быстро блокируется |

# 5. ПАНЕЛЬ УПРАВЛЕНИЯ

## 5.1 Основная панель

# 3x-ui (self-hosted)

### Назначение

Основная production-панель для управления VPN-инфраструктурой.

### Требования

- полностью self-hosted решение;
- отсутствие телеметрии;
- поддержка:
  - VLESS Reality;
  - Hysteria2;
  - AmneziaWG;
- управление пользователями;
- генерация ссылок автоматической подписки;
- управление несколькими inbound-подключениями;
- управление несколькими серверами;
- минимальное логирование;
- централизованное администрирование.

### Требования безопасности

- нестандартный URL панели;
- нестандартный порт;
- доступ к панели только по HTTPS;
- обязательная двухфакторная аутентификация;
- доступ только через:
  - VPN;
  - белый список IP-адресов;
- использование обратного прокси-сервера;
- fail2ban;
- отключение публичного доступа;
- регулярные обновления безопасности.

# 7. СЕРВЕРНАЯ ИНФРАСТРУКТУРА

## 7.1 Требования к серверам

Предпочтительные характеристики:

- VPS/VDS;
- NVMe SSD;
- канал от 1 Gbps;
- высокая связность с РФ;
- низкий ping;
- возможность быстрой смены IP;
- API для автоматизации;
- отсутствие anti-VPN ограничений.

---

## 7.2 Серверная инфраструктура: целевые регионы

### Финляндия (Хельсинки) — основной сервер

#### Назначение

Основной узел подключения для всех пользователей.

#### Ожидаемый ping из Казани

55–75 мс.

#### Требования к провайдеру

- KVM-виртуализация;
- NVMe SSD;
- канал от 1 Gbps;
- отсутствие anti-VPN ограничений в правилах использования;
- возможность смены IP по запросу;
- API для автоматизации.

#### Рекомендуемые провайдеры

- Hetzner Helsinki;
- Ishosting;
- Fornex.

---

### Латвия (Рига) — резервный сервер

#### Назначение

Горячий резервный узел и быстрый резервный канал.

#### Ожидаемый ping из Казани

50–65 мс.

#### Требования

- автоматическое переключение клиентов при деградации финского узла;
- характеристики аналогичны финскому узлу.

#### Рекомендуемые провайдеры

- Beget Riga;
- Timeweb Amsterdam/Riga;
- Ishosting.

---

## 7.3 Архитектура двух узлов

- оба сервера работают одновременно;
- 3x-ui используется для централизованного управления обоими узлами;
- ссылка автоматической подписки содержит оба сервера;
- клиент Hiddify автоматически выбирает сервер с минимальным ping;
- при отказе одного узла выполняется автоматическое переключение на резервный;
- ротация ключей и сертификатов синхронизируется между узлами.

---

## 7.4 Требования к независимости узлов

- разные провайдеры для финского и латвийского серверов;
- разные подсети и AS-номера;
- независимые домены для каждого узла;
- независимые TLS-сертификаты;
- отдельные UUID для каждого пользователя на каждом сервере.

---

## 7.5 Целевые показатели по узлам

| Параметр | Финляндия | Латвия |
|---|---|---|
| Ping из Казани | ≤75 мс | ≤65 мс |
| Доступность сервиса | ≥99% | ≥99% |
| Потери пакетов | ≤1–2% | ≤1–2% |
| Переключение при отказе | автоматическое | автоматическое |

---

## 7.6 Архитектура инфраструктуры

Инфраструктура должна поддерживать:

- несколько серверов;
- резервные IP;
- резервные домены;
- резервные протоколы;
- автоматическую замену точек подключения;
- быстрое масштабирование;
- замену серверов без downtime.
